Minecraft 基岩版行为包制作全流程教程

2026-06-29

本教程旨在为有一定 Minecraft 游戏基础、想深入了解或投身于基岩版 Add-on 开发的创作者提供系统化的指导。行为包是基岩版模组开发的核心，它决定了游戏内生物的行为、物品的属性、方块的功能以及底层的逻辑运行。

写代码和配置文件是一项需要极其严谨心态的工作。在制作行为包的过程中，哪怕只是漏掉了一个逗号，或者大小写拼错，都会导致整个行为包失效。请务必保持耐心，严格按照规范操作。

一、开发前的准备工作

在开始之前，搭建一个高效、规范的开发环境可以让你事半功倍。

1. 必备工具

  1. 代码编辑器：强烈推荐使用 Visual Studio Code。它是目前开发MC基岩版最主流的工具，拥有丰富的插件生态。
  2. Bedrock Edition Development：VS Code 插件。提供基岩版JSON代码的高亮、补全和错误提示。
  3. Blockbench：VS Code 插件。用于制作 3D 模型和导出基岩版文件（选装，主要用于资源包，但行为包关联生物时常用）。
  4. UUID 生成器：行为包需要唯一的身份标识，可以通过在线 UUID 生成器或 VS Code 的相关插件生成。

2. 游戏内设置

为了方便调试，建议专门建立一个测试创造地图，并在世界设置中开启以下选项：

  1. 测试版 API：如果你需要使用 JavaScript 脚本控制游戏逻辑，则必须开启此项。
  2. 内容日志文件：在游戏的“设置”>“全局资源”或“存储”侧边栏中，务必开启“内容日志历史记录”。当行为包报错时，这里会准确指出哪一行代码出了问题。

二、行为包的核心目录结构

基岩版的Add-on采用了数据驱动的设计理念，所有的机制都通过特定目录下的 JSON 文件来定义。一个标准且完整的行为包结构如下：


MyBehaviorPack/
├── manifest.json                  # 行为包的元数据与配置文件（核心）
├── pack_icon.png                  # 行为包的图标（256x256像素）
├── entities/                      # 存放生物行为文件的目录
│   └── custom_zombie.json         # 自定义生物或修改原版生物的文件
├── items/                         # 自定义物品的目录
│   └── ruby_sword.json
├── blocks/                        # 自定义方块的目录
│   └── ruby_ore.json
├── loot_tables/                   # 掉落物表目录
│   └── entities/
│       └── cow.json
├── recipes/                       # 合成表目录
│   └── ruby_sword_recipe.json
├── functions/                     # 函数文件目录（.mcfunction）
│   └── init.mcfunction
└── scripts/                       # Script API 脚本目录（可选）
    └── main.js

需要注意的是，文件夹的名称必须完全使用小写英文字母，且拼写不得有误，否则游戏引擎无法识别。

三、核心基础：编写manifest.json文件

manifest.json是行为包的身份证，没有它，游戏根本不会读取该文件夹。它定义了包的名字、描述、唯一UUID以及版本号。

在你的工作目录下新建一个名为manifest.json的文件，填入以下标准模板：


{
    "format_version": 2,
    "header": {
        "description": "这是我的第一个基岩版行为包",
        "name": "自定义行为包模板",
        "uuid": "4f3b2a1c-5d6e-7f8a-9b0c-1d2e3f4a5b6c",
        "version": [1, 0, 0],
        "min_engine_version": [1, 20, 0]
    },
    "modules": [
        {
            "description": "行为包核心模块",
            "type": "data",
            "uuid": "7a8b9c0d-1e2f-3a4b-5c6d-7e8f9a0b1c2d",
            "version": [1, 0, 0]
        }
    ],
    "dependencies": []
}
  1. format_version：定义清单文件的格式版本，通常为2。
  2. UUID：唯一通用识别码。注意，header和 modules 内的 UUID 绝对不能相同，且每一次制作新包都要重新生成。
  3. version：行为包的版本号。格式为[主版本, 次版本, 修订版本]，升级模组时需递增。
  4. min_engine_version：支持的最低游戏版本。它决定了你的包能调用哪个版本及以上的游戏组件。
  5. type：模块的类型。传统行为包的代码部分固定填"data"。若要启用脚本则需额外添加"script"模块。

四、实战演练一：修改原版生物行为（让牛掉落钻石）

接下来通过一个经典的改动来熟悉数据驱动的逻辑。我们将修改原版的“牛”，让它在被击杀时不再只掉落皮革和牛肉，而是有概率掉落钻石。

步骤1：创建掉落物文件

在行为包内依次建立文件夹：loot_tables/entities/。在此目录下创建一个名为cow.json的文件。

写入以下 JSON 代码：


{
    "pools": [
        {
            "rolls": 1,
            "entries": [
                {
                    "type": "item",
                    "name": "minecraft:beef",
                    "weight": 1,
                    "functions": [
                        {
                            "function": "set_count",
                            "count": {
                                "min": 1,
                                "max": 3
                            }
                        }
                    ]
                }
            ]
        },
        {
            "rolls": 1,
            "entries": [
                {
                    "type": "item",
                    "name": "minecraft:diamond",
                    "weight": 1,
                    "functions": [
                        {
                            "function": "set_count",
                            "count": {
                                "min": 1,
                                "max": 2
                            }
                        }
                    ]
                }
            ]
        }
    ]
}

我们对上述代码进行拆解：

  1. pools（掉落池）：游戏在计算掉落时会遍历每个池。上面有两个池，意味着牛肉和钻石的判定是独立的。
  2. rolls（抽取次数）：当前池被执行多少次。
  3. entries（条目）：池内包含的具体物品。name 指定物品ID，weight 代表权重。
  4. functions（函数修正）：set_count 用于控制数量。在上面的代码中，钻石的掉落数量被设定在 1 到 2 之间。

步骤2：覆写生物组件

若要完全控制牛的行为，可以在entities/ 文件夹下创建 cow.json。基岩版允许我们通过定义与原版同名的文件来覆盖原版行为。

你可以通过官方提供的Vanilla Add-ons模板获取原版牛的完整组件，修改其minecraft:loot 组件指向你自定义的掉落表：

"minecraft:loot": {
    "table": "loot_tables/entities/cow.json"}

五、实战演练二：从零创建自定义物品与合成表

仅仅修改原版内容是不够的，Add-on 的真正魅力在于创造全新的事物。下面我们将制作一把“红宝石神剑”。

1. 编写物品行为文件

在items/ 文件夹下创建ruby_sword.json：


{
    "format_version": "1.20.10",
    "minecraft:item": {
        "description": {
            "identifier": "custom:ruby_sword",
            "menu_category": {
                "category": "equipment",
                "group": "itemGroup.name.sword"
            }
        },
        "components": {
            "minecraft:display_name": {
                "value": "红宝石神剑"
            },
            "minecraft:icon": {
                "texture": "ruby_sword"
            },
            "minecraft:hand_equipped": true,
            "minecraft:max_stack_size": 1,
            "minecraft:damage": 8,
            "minecraft:weapon": {
                "on_hit_block_break": true
            },
            "minecraft:durability": {
                "max_durability": 2000
            },
            "minecraft:can_destroy_in_creative": false
        }
    }
}

我们对上述代码的关键组件进行解析：

  1. identifier（标识符）：命名空间必须符合命名空间:物品名 的格式。这里用custom:ruby_sword，避免与官方的 minecraft: 冲突。
  2. menu_category：决定了该物品在创造模式物品栏的哪一栏出现。
  3. minecraft:damage：赋予该物品基础攻击力。这里设定为 8 点（比下界合金剑还要高 1 点）。
  4. minecraft:durability：耐久度组件。max_durability 设为 2000。
  5. minecraft:icon：指向资源包中item_texture.json所定义的贴图键名。(注意：由于本篇主讲行为包，贴图需要配合资源包才能正常显示，否则在游戏中会显示为紫黑相间的方块，但其属性和功能依然生效)。

2. 编写合成表文件

为了能让生存模式的玩家获取这把武器，我们需要在recipes/文件夹下创建一个合成表文件ruby_sword_recipe.json：


{
    "format_version": "1.12.0",
    "minecraft:recipe_shaped": {
        "description": {
            "identifier": "custom:ruby_sword_recipe"
        },
        "tags": [
            "crafting_table"
        ],
        "pattern": [
            " R ",
            " R ",
            " S "
        ],
        "key": {
            "R": {
                "item": "minecraft:ruby"
            },
            "S": {
                "item": "minecraft:stick"
            }
        },
        "result": {
            "item": "custom:ruby_sword",
            "count": 1
        }
    }
}

一些逻辑说明：

  1. recipe_shaped表示有序合成。如果要无序合成则使用 recipe_shapeless。
  2. pattern 直观地模拟了 3 × 3 工作台的网格。空格代表不放任何东西，R 和 S 分别对应下面key绑定的物品。
  3. result 则是产出的结果。

六、进阶开发：引入 Script API

传统的 JSON 行为包采用的是数据声明式的逻辑，无法处理极其复杂的动态条件（比如：判断玩家的血量是否低于 20% 从而触发雷击）。为了解决这个问题，现代基岩版（1.19+）引入了基于 JavaScript 的 Script API。

如果你想让你的行为包具有更高的自由度，可以通过以下步骤引入脚本。

1. 修改manifest.json 以支持脚本

你需要在原有的manifest.json 中，在modules 数组里追加一个 script 类型的模块，并在 dependencies 中声明你需要的 API 库。

修改后的manifest.json 部分片段如下：


"modules": [
        {
            "description": "行为包核心模块",
            "type": "data",
            "uuid": "7a8b9c0d-1e2f-3a4b-5c6d-7e8f9a0b1c2d",
            "version": [1, 0, 0]
        },
        {
            "description": "JavaScript 脚本模块",
            "type": "script",
            "language": "javascript",
            "uuid": "3b4c5d6e-7f8a-9b0c-1d2e-3f4a5b6c7d8e",
            "version": [1, 0, 0],
            "entry": "scripts/main.js"
        }
    ],
    "dependencies": [
        {
            "module_name": "@minecraft/server",
            "version": "1.10.0"
        }
    ]

2. 编写脚本文件

在scripts/ 目录下新建 main.js。我们来实现一个简单的逻辑：当玩家敲击方块时，在公屏打印一条欢迎语，并赋予玩家短时间的加速效果。 

import { world, system, EffectType } from "@minecraft/server";

// 监听玩家破坏/点击方块的事件
world.afterEvents.playerBreakBlock.subscribe((eventData) => {
    const player = eventData.player;
    const block = eventData.block;

    // 向全服或当前玩家发送聊天栏消息
    player.sendMessage(`你刚刚破坏了一个系统方块: ${block.typeId}`);

    // 给玩家施加一个速度效果 (持续100 tick，即5秒，强度为2)
    system.run(() => {
        player.addEffect("speed", 100, {
            amplifier: 1,
            showParticles: true
        });
    });
});

为什么脚本开发需要谨慎？

JavaScript 是图灵完备的编程语言。这意味着，错误的循环逻辑（例如死循环while(true)）会直接导致整个游戏客户端或服务器卡死崩溃。在使用 Script API 时，必须严格遵循事件订阅机制，并注意对象的生命周期。

七、调试、测试与常见错误排查

编写好代码后，直接将其放入游戏中进行检验是唯一的试金石。

1. 如何导入游戏

最推荐的方法是在开发阶段目录内导入。不要把行为包直接打包，而是将你的整个开发文件夹（例如MyBehaviorPack）复制到基岩版的专用开发目录下：%localappdata%\Packages\Microsoft.MinecraftUWP_8wekyb3d8bbwe\LocalState\games\com.mojang\development_behavior_packs\。放到这里的文件夹会自动实时同步到游戏内，修改代码后只需退出并重新进入世界即可刷新，无需反复导入。

还有一种方法就是在发布阶段打包。将文件夹内的所有文件（注意：是全选文件，而不是选择外层文件夹）压缩为.zip 格式，然后将后缀名直接改为.mcpack。双击该文件，游戏便会自动启动并导入。

2. 核心调试技巧与日志分析

很多初学者最常遇到的问题就是：“为什么我把行为包激活了，进游戏却什么都没有发生？”

此时，请按以下步骤理性排查，切忌盲目乱猜：

  1. 检查 Content Log（内容日志）窗口：如果在进入世界时弹出了一个满是红字或黄字的窗口，这说明游戏引擎读取到了你的包，但语法有误。
  2. 检查 UUID 冲突：如果你复制了别人的manifest.json却忘记修改 UUID，游戏会直接忽略你的包，且不会报任何错误。
  3. 大写与小写敏感：基岩版对文件的路径和文件名大小写极其敏感。如果在 JSON 里写了"table": "loot_tables/entities/Cow.json"，而实际文件名是cow.json（小写），在 Windows 系统上可能偶尔能跑通，但在移动端（Android/iOS）或服务器（Linux）上必定失效。

八、结语

制作 Minecraft 基岩版行为包是一个将创意转化为现实的过程。从最基础的修改原版生物属性，到利用 JSON 构建全新的方块物品，再到利用 Script API 编写复杂的微型玩法模块，它的上限极高。

初学者往往会被枯燥的代码和频繁的报错劝退，但只要你掌握了规范的目录结构，学会利用 VS Code 插件进行语法纠错，并养成了看内容日志的习惯，你会发现 Add-on 开发具有极强的逻辑美感。多参考官方公开的原版模板文件，多动手实践，是通往高阶创作者的唯一路径。
